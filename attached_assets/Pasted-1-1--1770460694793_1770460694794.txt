1) На что назначать группы в первую очередь
1) Материал — “группа видимости материала”
Это главный объект.
•	В паспорте материала: Группа видимости (по умолчанию “Базовая”).
•	Если группа ≠ “Базовая” → материал виден только участникам этой группы:
o	каталог/поиск/витрины/прямая ссылка
o	скачивание файла/просмотр страницы
o	экспорт реестра/отчёты (только то, что доступно)
2) Подраздел — “дефолтная группа подраздела”
Чтобы не назначать группу руками на каждом материале.
•	В подразделе: Группа по умолчанию (например “Финансы”).
•	При создании материала в подразделе группа проставляется автоматически.
•	Смена дефолта не меняет старые материалы.
2) Где группы должны “влиять” (точки применения, чтобы не было дыр)
Чтобы механизм был непробиваемым, проверка группы должна стоять везде, где пользователь может получить доступ:
1.	Списки и выборки
каталог, витрины, поиск, “мои материалы”, очереди — показывают только доступные по группе материалы.
2.	Карточка материала
прямая ссылка должна проверять доступ по группе и выдавать “нет доступа” без утечки содержимого.
3.	Скачивание/просмотр контента
эндпоинт файла/PDF/страницы тоже проверяет группу (иначе утечка через прямой URL).
4.	Экспорт в Excel / отчёты / дашборды
выгрузка и метрики должны учитывать доступ (иначе утечка через отчёт).
5.	Подписки и email
перед отправкой писем по подписке/ознакомлению/пересмотрам — проверяем, что получатель имеет доступ по группе.
3) Как назначать группы “операционно”, чтобы это работало без хаоса
Кто назначает группу материалу
•	Владелец/Заместитель/Администратор и Автор.
•	Изменение группы — через новую версию (как у вас).
Важная защита при смене группы
При смене группы на более закрытую показывать предупреждение:
•	“После публикации доступ потеряют: N пользователей / группы …”
•	И напоминание о вашем правиле: потеряют доступ ко всем старым версиям тоже.
4) Что ещё привязать к группам 
A) “Ознакомление” и “новые сотрудники”
У вас уже логика завязана на “кому доступно” — это идеально ложится на группы:
•	Требуется ознакомление → рассылка всем, кому доступен материал (т.е. всем участникам его группы).
•	Обязательно для новых сотрудников → новый пользователь получает список только тех mandatory-материалов, которые доступны ему по его группам.
B) Видимость обсуждений/комментариев
Если у материала обсуждения “ограничены”, вместо “роли/группы AD” использовать ваши портальные группы:
•	“обсуждения видят только группы: …”
(Просто, и не требует AD.)
C) Главная страница по логике не меняется от групп 
5) может быть несколько групп материал
Материал доступен, если пользователь состоит хотя бы в одной из групп.

6) Минимальный чек-лист, чтобы группы “заработали” уже завтра
1.	Сделать поле material.visibility_group_id эффективным и использовать его в проверках доступа.
2.	Фильтровать по нему все списки/поиск/витрины.
3.	Проверять по нему доступ на скачивание/просмотр контента.
4.	Добавить subsection.default_visibility_group_id для автоподстановки.
5.	В админке — CRUD групп и участников + системная “Базовая”.
